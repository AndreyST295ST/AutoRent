Нефункциональные требования

Сервис по аренде автомобилей

1. Производительность (Performance)
    Время загрузки страниц каталога и профиля: не более 2 секунд при нормальной нагрузке.

    Поиск автомобилей с фильтрацией (по классу, цене, КПП): не более 3 секунд.

    Система должна выдерживать одновременную работу не менее 1000 пользователей и обрабатывать до 100 бронирований в минуту в пиковые часы.

2. Надежность и доступность

    Доступность системы: не менее 99,5% в месяц.

    Время восстановления после сбоя (RTO): не более 2 часов.

    Резервное копирование базы данных – ежедневно, хранение копий не менее 30 дней.

    Корректная обработка ошибок: пользователь получает понятные сообщения, система логирует все исключения.

3. Безопасность

    Аутентификация на основе JWT или OAuth2 с разграничением прав для ролей.

    Пароли хранятся в зашифрованном виде (bcrypt).

    Все данные передаются по протоколу HTTPS (TLS 1.3).

    Платёжные данные не хранятся в системе – используется токенизация через внешний платёжный шлюз (PCI DSS).

    Логирование всех критических действий (вход, изменения прав, финансовые операции).

4. Масштабируемость

    Поддержка горизонтального масштабирования приложений и базы данных для роста нагрузки.

    Возможность увеличения автопарка без ограничений на количество автомобилей.

    Добавление новых функций без изменения основной архитектуры.

5. Удобство использования

    Интуитивно понятный интерфейс: бронирование автомобиля должно занимать не более 8 кликов.

    Адаптивный дизайн для корректного отображения на ПК, планшетах и мобильных устройствах.

    Наличие подсказок, понятных иконок и визуальной обратной связи при действиях.

6. Совместимость

    Поддержка современных браузеров: Chrome, Firefox, Safari, Edge (последние 2 версии).

    Возможность интеграции с внешними сервисами: платёжные шлюзы (ЮKassa, Stripe), гео‑сервисы (карты), SMS‑шлюзы для уведомлений.

7. Поддерживаемость и расширяемость

    Код должен быть документирован и соответствовать стандартам разработки (например, следование REST, комментирование API через OpenAPI).

    Архитектура должна позволять добавлять новые типы тарифов, категории автомобилей, способы оплаты без изменения существующих модулей.

8. Локализация и интернационализация

    Основной язык интерфейса – русский. Возможность добавления других языков без изменения кода.

    Форматы дат, времени, чисел и валют должны соответствовать локальным стандартам пользователя.

9. Логирование и мониторинг

    Ведение журнала действий пользователей для аудита и отладки.

    Мониторинг состояния серверов, базы данных и ключевых сервисов с отправкой уведомлений администраторам при критических сбоях.
